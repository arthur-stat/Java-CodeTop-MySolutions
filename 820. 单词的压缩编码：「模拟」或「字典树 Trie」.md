题目简述：

> 单词数组 `words` 的 **有效编码** 由任意助记字符串 `s` 和下标数组 `indices` 组成，且满足：
>
> - `words.length == indices.length`
> - 助记字符串 `s` 以 `'#'` 字符结尾
> - 对于每个下标 `indices[i]` ，`s` 的一个从 `indices[i]` 开始、到下一个 `'#'` 字符结束（但不包括 `'#'`）的 **子字符串** 恰好与 `words[i]` 相等
>
> 给你一个单词数组 `words` ，返回成功对 `words` 进行编码的最小助记字符串 `s` 的长度 。

题目链接：[820. 单词的压缩编码](https://leetcode.cn/problems/short-encoding-of-words/)

# 模拟

## 思路

这个是最容易想到的思路，直接模拟即可。

不难知道，当 $m>n$，则长为 $n$ 的单词有可能被压缩编码在长为 $m$ 的单词中，而长为 $m$ 的单词绝无可能被压缩编码在长为 $n$ 的单词中。

因此，为了尽可能地压缩编码，我们将单词数组 `words` 按长度重排序为倒序。使用一张哈希表直接记录所有单词的可能的压缩编码单词，遍历按长度倒序排序后的单词数组 `words_sorted`，对于一个新单词 `word`：

1. 检查 `word` 是否已位于哈希表中，如果已位于哈希表，则 `word` 能产生的所有有效压缩编码单词都已经被记录在哈希表中了，于是跳过该单词、考察下一个单词并重复该步骤，这点由步骤 2 所保证；
2. 至此已可以明确 `word` 不位于哈希表中，因此没有已知的单词能压缩编码得到 `word`，于是将 `word` 添加到助记字符串 `s` 中，然后将 `word` 能产生的所有可能的压缩编码单词添加至哈希表中，随后继续考察下一个单词，回到步骤 1。

该方案简单暴力，空间复杂度较高，但至少行之有效。

## 代码

记总单词数为 $n$，则算法的时间复杂度为 $O\big(n\log n+\sum\mathrm{len}^2(word_i)\big)$，空间复杂度为 $O\big(\sum\mathrm{len}^2(word_i)\big)$。

```java
class Solution {
    public int minimumLengthEncoding(String[] words) {
        Arrays.sort(words, (a, b) -> Integer.compare(b.length(), a.length()));
        Set<String> set = new HashSet<>();
        String s = "";

        for (String word : words) {
            if (!set.contains(word)) {
                int len = word.length();
                for (int i = 0; i < len; i++) {
                    // 这里 if-break 是为了提前结束，但在测试中这样做并没有为算法显著提速
                    if (!set.add(word.substring(i, len))) break;
                }
                s += word + "#";
            }
        }

        return s.length();
    }
}
```

# 翻转 + 字典树 Trie

## 思路

上述暴力模拟的思路整体上是可行的，问题在于用哈希表存储了所有可能的被压缩编码的单词，这导致复杂度可观地增加了。

考虑更合适的数据结构，我们不应使用哈希表来存储所有可能的压缩编码单词；对于本问题，更适合的数据结构是字典树 Trie。这是因为，对于单词 `ichika`，如果考虑哈希表，则需要存储 `ichika`、`chika`、`hika`、`ika`、`ka` 以及 `a` 这几个压缩编码字符串；而如果考虑字典树 Trie，则只需要存储 `i - c - h - i - k - a` 这一条链。显然，空间使用效率大大提高了。

要注意，在使用字典树 Trie 解决本问题时，需要翻转以逆序存储压缩编码字符串。这是因为，标准的 Trie 是前缀树，而此处显然地我们需要考虑后缀树——`hika` 被包含在 `ichika` 中，但若是考虑前缀树，我们是无法高效判断 `hika` 是 `ichika` 能被压缩编码的，因为二者并没有相同前缀。然而，注意到二者有相同后缀，因此如果我们可以考虑后缀树，这样就能方便地判断出 `hika` 能够被 `ichika` 压缩编码。

换言之：**<u>本问题中的所谓压缩编码，实际上就是后缀子串</u>**！！所以，本问题实质上就是为字典树这一数据结构而生的。

## 代码

算法的时间复杂度为 $O(n\log n+S)$，空间复杂度为 $O(S)$，其中 $S$ 为所有单词的长度总和。

> **在通常的 Trie 实现中，会使用一个标记 `isEnd` 显式地标记当前节点是否为某个单词的尾节点。**对于本问题，我们可以适当「偷懒」，因为本问题中我们<u>无需区分一条路径上的各个单词</u>——我们只关心最长的那个单词，剩余的单词都可以被视为最长的那个单词的 “压缩编码”。

```java
class Solution {
    public int minimumLengthEncoding(String[] words) {
        Arrays.sort(words, (a, b) -> Integer.compare(b.length(), a.length()));
        Trie trie = new Trie();
        int res = 0;
        for (String word : words) {
            if (!trie.build(word)) res += word.length() + 1;
        }
        return res;
    }

    class Trie {

        Node dummy;

        Trie() {
            dummy = new Node('\0');
        }

        boolean build(String word) {
            return build(dummy, word, word.length() - 1);
        }

        boolean build(Node node, String word, int i) {
            if (i < 0) return true;

            char c = word.charAt(i);
            if (node.contains(c)) {
                return build(node.get(c), word, i - 1);
            } else {
                Node next = new Node(c);
                node.put(next);
                build(next, word, i - 1);
                return false;
            }
        }

        class Node {

            char c;
            // boolean isEnd;
            Map<Character, Node> map;

            Node(char c) {
                this.c = c;
                // isEnd = false;
                map = new HashMap<>();
            }

            boolean contains(char c) {
                return map.containsKey(c);
            }

            boolean isEmpty() {
                return map.isEmpty();
            }

            Node put(Node node) {
                return map.put(node.c, node);
            }

            Node get(char c) {
                return map.get(c);
            }
        }        
    }
}
```
